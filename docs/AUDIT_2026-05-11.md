## PART 2 — VOICE AGENT BUG AUDIT

Baseline is OPTION-A-FINAL (`utils/claudeService.js` is 828 lines).

### BUG A — Availability overflow past closing hours

**Symptom**: Agent offered a 4-hour Karaoke slot at 10pm when the business closes at midnight (would extend to 2am).

🟡 WARN — multiple co-located issues. The engine itself enforces the close-buffer correctly (good), but the prompt advertises a `LATEST START` that the agent reads as universally valid for *any* duration the customer picks.

#### Where the failing logic lives

**Primary surface — `utils/claudeService.js` lines 67–99 (`deriveServiceConstraints`):**

```js
const minDur = durations.length > 0 ? durations[0] : (s.duration_minutes ? Number(s.duration_minutes) : null);
…
if (serviceClose && minDur) {
  const [h, m] = serviceClose.split(":").map(n => parseInt(n, 10));
  let closeMin = h * 60 + m;
  if (closeMin === 0) closeMin = 24 * 60;
  const latestMin = closeMin - minDur;
```

`latestStart` is computed against the **smallest** bookable duration only. The prompt then renders this as a top-level field on the service line (`buildServiceLine`, line 138):

```js
const latestStartStr = constraints.latestStart
  ? `LATEST START: ${constraints.latestStart}${constraints.serviceClose ? ` (service closes ${constraints.serviceClose})` : ""}`
  : null;
```

For Karaoke with rate-rule durations 120/180/240 and close at midnight, the prompt says `BOOKABLE DURATIONS: 2hr / 3hr / 4hr | LATEST START: 22:00 (service closes 00:00)`. The agent reads "you can start at 22:00" and "duration 4hr is allowed" as independently true facts. There's no per-duration latest start.

**Secondary surface — `utils/voiceContext.js` lines 400–429 (`fmtSvc`):** The voice path builds an analogous service line but does **not** include any `LATEST START` hint at all. It only shows `min N min`/`max N min` and `hours: …`. So in the voice path the agent had even less to go on:

```js
const minDur = (slotInt && minSlots) ? `min ${slotInt * minSlots} min` : null;
const maxDur = (slotInt && maxSlots && maxSlots > 1)
  ? `max ${slotInt * maxSlots} min`
  : null;
```

#### Engine behaviour (the hard backstop)

`utils/availabilityEngine.js` lines 145–151 *does* enforce the right thing:

```js
// VOICE-FIX-1 (Bug 1): Slots must end before close, not just start before close-stepMin.
…
const closeBufferMin = Math.max(stepMin, durationMin);
```

And lines 281–302 cap windows by `serviceHoursWindows` and `slotFallsWithinServiceHours()` uses full `durationMin` — confirmed in line 300–301:

```js
const end   = start + durationMin;
return serviceHoursWindows.some((w) => start >= w.start_minute && end <= w.end_minute);
```

So `check_availability` would NOT have returned 22:00 as a 4-hour-bookable slot.

#### routes/ai.js validation on slots before they leave the API

`routes/ai.js` `handleAction` `case "create_booking"` lines 938–986 enforces the `slotConfirmationCache` gate AND a defensive `hasConflictingBooking` check, plus availability re-derives via the engine. Server side is fine.

#### Was `check_availability` actually called?

🔴 CRITICAL — likely not. The agent emitted a slot from prompt context. Evidence:

- The "exception" carve-out in `claudeService.js` lines 542–544 explicitly gives the agent permission to skip the tool when the asked-for time is "CLEARLY outside the service's operating window":

  ```text
  The only exception: if the customer asks for a time CLEARLY outside the service's operating window
  (e.g. "Karaoke at 6am" when Karaoke runs 8pm–midnight), you may explain the rule first WITHOUT
  calling check_availability, then offer the nearest valid window
  ```

  The agent inverted that clause: 22:00 is *inside* the operating window, so the same prompt that authorises skipping check_availability for "outside" times implicitly also encourages the agent to answer from prompt for "inside" times.

- `LATEST START: 22:00` reads as authoritative — the agent's heuristic-fast-path then proposes 22:00 with a 4-hour duration without actually calling `check_availability`.

- Confirming this would require call-trace inspection. **UNKNOWN — would need the Anthropic call log for that booking turn to confirm whether `check_availability` was emitted.**

#### Reproduction conditions
1. Tenant has Karaoke service with `service_hours` configured 20:00–00:00 every day, rate rules `min_duration_mins` 120/180/240.
2. Customer asks "I want 4 hours of karaoke tonight starting at 10pm."
3. Agent reads `BOOKABLE DURATIONS: 2hr / 3hr / 4hr | LATEST START: 22:00`.
4. Agent answers conversationally without emitting `check_availability` (or, if it does, the engine returns no slot at 22:00 for 4hr — and the agent ignores that and proposes 22:00 anyway).

#### 2 alternative fix surfaces (DO NOT IMPLEMENT)

| Surface | Tradeoff |
|---|---|
| Prompt rule change in `buildServiceLine` (claudeService.js 102–152) — emit a `LATEST START` per duration: `LATEST START: 22:00 (2hr) / 21:00 (3hr) / 20:00 (4hr)` | Pro: zero engine change, instant fix for the "answer from prompt" failure mode. Con: lengthens every service line; brittle if rate rules change shape; doesn't constrain bad behaviour, only documents it. |
| Tool schema change — make `check_availability` mandatory on every booking proposal: backend rejects any `create_booking` whose `start_time + duration > slotConfirmationCache.lastCheckAt` window for that service+date. Hard server-side gate. | Pro: turns the issue into a 500 instead of a customer-visible bad booking. Con: adds latency (forces a tool call before each propose), and the cache TTL of 5 min is short for some flows. |

Backend-safety-net option (third): tighten `routes/ai.js` `handleAction.create_booking` to re-call the engine for `(serviceId, date, durationMin)` and refuse if the slot isn't returned, even on cache hit — currently the cache only tracks `time` (line 35: `slots.map(s => ({ time: s.time, resource_id: s.resource_id ?? null }))`), which is duration-blind.

### BUG B — Agent offered to book over an existing confirmed booking

**Symptom**: Customer had Karaoke confirmed at 9pm. Agent agreed to book Karaoke at 8pm for them (overlapping conflict).

🔴 CRITICAL — matches the "answer-from-prompt" failure mode of Bug A but with customer-history evidence.

#### Status filter on the engine

`utils/availabilityEngine.js` blocks BOTH pending AND confirmed (lines 405 and 439, identical clauses):

```js
LEFT JOIN bookings b
  ON b.tenant_id = $${tenantIdIdx} AND b.status IN ('pending','confirmed')
 AND b.booking_range && tstzrange(slot_start, slot_end, '[)')
 AND b.deleted_at IS NULL
```

This is correct — engine would have refused 8pm if it saw the 9pm confirmed booking (a 4-hour 8pm booking ending at 12am overlaps a 9pm booking).

#### Defensive secondary check

`routes/ai.js` lines 65–84 (`hasConflictingBooking`) uses the same `status IN ('pending','confirmed')` clause. Booking insert path: `routes/bookings/create.js` line 318–333 calls `checkConflicts({…})` from `utils/bookings.js` (not read here, but flagged for follow-up — UNKNOWN whether `checkConflicts` covers the same statuses; would need to read `utils/bookings.js`).

The agent sees the 9pm Karaoke booking. It either (a) didn't realise 8pm + 4hr overlaps 9pm, or (b) didn't call `check_availability` and just proposed 8pm because it "looked free".


#### What's missing

- No prompt rule says "before proposing any booking, sanity-check it against the customer's UPCOMING BOOKINGS for this service for time-overlap". The closest rule is the duplicated AVAILABILITY one at `claudeService.js` line 606: `AVAILABILITY: Always call check_availability before confirming any slot.` — but this isn't behaviourally enforced.
- The check_availability tool, when called with no `resource_id` (line 627–648 of `routes/ai.js`), fans out across all linked resources of that service. The result includes structured `ownership: "YOUR" | "OTHER" | "BUSY"` tags (line 806–808), so the agent *would* see "Sim 1 BOOKED (YOUR existing booking)" if it had called the tool. 2. Customer says "Can you book me Karaoke at 8pm tonight, 4 hours?"
3. Agent answers from prompt — sees Karaoke is in `BOOKABLE DURATIONS: 2hr / 3hr / 4hr` and `LATEST START: 22:00`, sees 8pm is in business hours. Agrees without calling check_availability, or calls it, sees no slots, and ignores result.

| Surface | Tradeoff |

Con: relies on the LLM doing arithmetic correctly across timezone strings. |

Cancel or move that one first?"`. | Pro: deterministic. Con: requires an extra DB round-trip per `create_booking`; small. |

Tool-schema option: extend `check_availability`'s response so any "BUSY (YOUR…)" slot also surfaces a structured `your_existing_booking_id` field that the agent must reference before proposing — making it harder to ignore the overlap.

```js
const ninetyDaysAgo = now - 90 * 24 * 60 * 60 * 1000;
```
